                Cortex Scripts (4)
                

                    
ExtractK8sContainerEscapeIOCs
Analyzes XDR issue fields to extract IOCs (container ID, namespace, node FQDN, process name/SHA256, container image). Determines incident severity (Critical/High/Medium/Low) based on attack indicators (Spring4Shell, webshell, container escape, credential theft). Detects Spring4Shell patterns and webshell deployment. Outputs K8sEscape.* context used by all downstream scripts and playbooks. Writes triage report to k8scontainerescapeiocs issue field.

                    
InvokeK8sContainmentLambda
Invokes the AWS Lambda containment function from Cortex XSIAM (GCP-hosted, no boto3). Uses pure SigV4 signing (hmac/hashlib) to authenticate: first STS AssumeRole to get temporary credentials scoped to the lambda-invoker role, then Lambda Invoke with the containment action payload. Supports 7 actions: collect_evidence, network_isolate, revoke_rbac, scale_down, cordon_node, delete_pod, full_containment. Dual-mode: direct invocation if no assume_role_arn. Outputs K8sContainment.* context and writes to k8scontainmentenrichment issue field.

                    
K8sForensicAnalysis
Performs deep forensic analysis: CVE enrichment (Spring4Shell CVE-2022-22965, Spring Cloud CVE-2022-22963 with CVSS scores), MITRE ATT&CK kill chain mapping (9 techniques from T1190 Initial Access to T1530 Collection), and container escape indicator detection (nsenter, mount, chroot, /proc/1/root, IMDS, docker.sock, etc.). Generates 5 XQL forensic queries: causality chain, file operations, network connections, container escape patterns, and credential access. Outputs K8sForensic.* context (including XQLQueries array) and writes to k8sforensicanalysis issue field.

                    
K8sSearchSimilarEvents
Generates cross-tenant threat hunting XQL queries to determine blast radius and detect lateral movement. Produces targeted searches (same process on other nodes, same binary SHA256 across endpoints, same container image, alerts in same namespace) and broad threat hunts (webshell drops on all K8s nodes, container escape patterns, IMDS credential theft). Targeted queries require specific IOCs from K8sEscape context; broad hunts always run. Outputs K8sSimilar.* context and writes to k8ssearchsimilarevents issue field.

                

            


            

                Cortex Playbooks (3)
                

                    
Containment Playbook (10 tasks)
Automated incident response for K8s container escape. Triages the XDR issue to extract IOCs, collects forensic evidence via Lambda (pod details, logs, events, RBAC audit, node status), then gates on severity: Critical/High with Spring4Shell indicators proceeds automatically, otherwise requests operator approval. Executes full containment sequence: deny-all NetworkPolicy, cluster-admin ClusterRoleBinding deletion, deployment scale-down to 0, node cordoning, force pod deletion. Final verification step re-collects evidence to confirm containment.

                    
Forensic Analysis Playbook (9 tasks)
Deep investigation of a container escape incident. Triages the issue, then runs K8sForensicAnalysis for CVE enrichment (Spring4Shell, CVSS 9.8), MITRE ATT&CK mapping, and XQL query generation. Automatically executes 5 XQL queries via xdr-xql-generic-query: causality chain reconstruction, suspicious file operations (webshell drops, config reads), network connections (IMDS, C2, K8s API), container escape patterns (nsenter, chroot, mount), and credential access attempts (SA tokens, AWS IMDS, kubeconfig). Concludes with live evidence collection via Lambda.

                    
Search Similar Events Playbook (8 tasks)
Threat hunting to determine if the attack has spread beyond the initially compromised node. Generates targeted and broad XQL search queries. Automatically executes 3 broad threat hunts via xdr-xql-generic-query: webshell file drops (.jsp) across all K8s nodes, container escape patterns (nsenter, /proc/1/root, chroot) on all endpoints, and IMDS credential theft (169.254.169.254) from containers. Targeted IOC queries (lateral movement by process name, binary hash hunt, container image blast radius, namespace alert correlation) are stored in the issue field for manual Query Center execution. Analyst reviews results and decides: escalate (similar events found) or close (no spread detected).

K8s Container Escape Demo

                    
This demo illustrates a full attack chain exploiting Kubernetes misconfigurations, from initial RCE to cluster takeover, and shows how Cortex XDR detects and responds to each stage.

                


                
                

                    

                        

                        
                        

                             Attack flow
                             Detection
                             Response
                        

                    

                


                

                    

                        

                            1
                            Infrastructure Setup
                        

                        
Deploy an EKS cluster on AWS with Terraform using admin credentials (one-time bootstrap). Terraform creates a dedicated dashboard IAM User with permanent Access Key credentials and a scoped dashboard operator IAM Role with least-privilege permissions (EKS, ECR, Lambda, VPC, IAM). After deployment, use the dashboard user's permanent credentials — the dashboard automatically assumes the operator role via STS AssumeRole. Terraform state is stored locally (no S3 backend needed).

                        
Build a vulnerable Spring Boot application (CVE-2022-22965) as a Docker image and push it to ECR. Deploy it as a privileged pod with dangerous security settings:

                        
    
                            
  • privileged: true - full access to host devices
    • 
                            
  • hostPID: true - visibility into host processes
    • 
                            
  • hostPath: / - read/write access to host filesystem
    • 
                            
  • cluster-admin ServiceAccount - full K8s API access
    • 
                        

                    

                


                

                    

                        

                            2
                            Attack Chain
                        


                        

                            
Step 1: Spring4Shell RCE (CVE-2022-22965)

                            
Exploit the Spring4Shell vulnerability to manipulate Tomcat's AccessLogValve via the classLoader. This writes a JSP webshell to the application directory, providing remote command execution on the container.

                            
MITRE: T1190 (Exploit Public-Facing Application) / T1505.003 (Web Shell)

                        


                        

                            
Step 2: Container Escape

                            
Leverage the pod's privileged configuration to escape the container:

                            
    
                                
  • hostPID - access host process tree via /proc
    • 
                                
  • hostPath / - read host filesystem (kubelet creds, SSH keys, etc.)
    • 
                                
  • nsenter - enter host namespaces to execute commands as root on the node
    • 
                                
  • IMDS - access AWS Instance Metadata to steal IAM credentials
    • 
                            

                            
MITRE: T1611 (Escape to Host) / T1552 (Unsecured Credentials)

                        


                        

                            
Step 3: Cluster Takeover

                            
Steal the pod's cluster-admin ServiceAccount token via the webshell, then deploy a kubectl attack script inside the container to enumerate the entire cluster: namespaces, pods, secrets, nodes, RBAC bindings.

                            
MITRE: T1078.004 (Cloud Accounts) / T1552.007 (Container API)

                        


                        

                            
Step 4: K8s Vulnerability Scanning

                            
Run K8s reconnaissance tools inside the container: permission enumeration (auth can-i), RBAC scanning, service discovery, pod security posture audit. Simulate deepce, kube-hunter, and peirates execution. Probe kubelet (10250) and etcd (2379) APIs from the host network namespace.

                            
MITRE: T1613 (Container Discovery) / T1610 (Deploy Container)

                        


                        

                            
Step 5: Deploy Malware

                            
Download and execute known malware samples: WildFire test ELF, reverse shell script, cryptominer simulation. Deploy deepce container escape scanner. Harvest credentials from /etc/shadow and kubelet kubeconfig.

                            
MITRE: T1105 (Tool Transfer) / T1059.004 (Unix Shell) / T1587.001 (Malware)

                        


                        

                            
Step 6: Lateral Movement

                            
Attempt to spread beyond the compromised node: SSH scanning of the VPC subnet, deploy a rogue privileged pod in kube-system, steal AWS IAM credentials via IMDS, access secrets in other namespaces, probe kubelet on other cluster nodes.

                            
MITRE: T1021 (Remote Services) / T1550.001 (Application Access Token) / T1530 (Cloud Storage)

                        

                    

                


                

                    

                        

                            3
                            Cortex XDR Detection & Response
                        

                        
Cortex XDR agent deployed on the cluster detects each stage of the attack:

                        
    
                            
  • Local Threat Detected - JSP webshell detected as malware on creation
    • 
                            
  • Anti-Webshell Protection - detects webshell execution patterns
    • 
                            
  • Container Escape Protection - detects privileged container escape via mount
    • 
                            
  • Behavioral Threat Protection - detects suspicious causality chains
    • 
                            
  • Credential Gathering - detects /etc/shadow and kubeconfig access
    • 
                            
  • K8s Vulnerability Scanning - detects recon tools (deepce, kube-hunter, peirates)
    • 
                            
  • Reverse Shell Protection - detects reverse shell connection attempts
    • 
                            
  • Cryptominer Protection - detects cryptominer-like activity
    • 
                        

                        
A prevention policy (rules + profiles) is imported on the Cortex tenant and assigned to the EKS endpoint group for K8s-specific protection.

                            4
                            Playbook Authentication Flow
                        

                        
Cortex XSIAM runs on GCP — no AWS SDK or implicit credentials available. The containment script (InvokeK8sContainmentLambda) uses pure SigV4 signing with hmac/hashlib for all AWS API calls.


                        

                            
Step 1: Operator Credentials

                            
The playbook passes the cortex-playbook-user permanent Access Key (created by Terraform) to the script. These credentials never expire and have a single permission: sts:AssumeRole on the lambda-invoker role.

                        


                        

                            
Step 2: STS AssumeRole (SigV4-signed)

                            
The script signs a POST request to sts.<region>.amazonaws.com using the operator credentials. AWS STS returns temporary credentials (AccessKeyId, SecretAccessKey, SessionToken) scoped to the lambda-invoker role.

                        


                        

                            
Step 3: Lambda Invoke (SigV4-signed)

                            
The script signs a second request to lambda.<region>.amazonaws.com using the temporary credentials from Step 2. The Lambda is invoked with the containment action payload (action, cluster, namespace).

                        


                        

                            
Step 4: Lambda → EKS API

                            
The Lambda function authenticates to EKS using an STS presigned URL with the x-k8s-aws-id header, then executes Kubernetes API calls for containment actions (NetworkPolicy, RBAC, scale down, cordon, etc.).

                        


                        
Dual-mode: if assume_role_arn is not provided, Step 2 is skipped and the Lambda is invoked directly with the operator credentials (direct mode).

CVE-2022-22965 — Spring4Shell (CVSS 9.8)

                    

                        

                            CRITICAL
                            Spring Framework 5.3.0–5.3.17
                        

                        
Remote Code Execution via ClassLoader manipulation in Spring MVC parameter binding. Allows unauthenticated attackers to deploy webshells on Tomcat servers.

                        

                            Fix: Upgrade to Spring Framework 5.3.18+ or Spring Boot 2.6.6+
                        

                    

                


                

                    
Kubernetes Misconfigurations (deployment.yaml)


                    

                        

                            CRITICAL
                            privileged: true
                        

                        
securityContext:
  privileged: true    # Full host kernel access

                        
Fix: privileged: false + allowPrivilegeEscalation: false

                    


                    

                        

                            CRITICAL
                            hostPID: true
                        

                        
spec:
  hostPID: true       # Access host process namespace → nsenter escape

                        
Fix: Remove hostPID or set to false

                    


                    

                        

                            HIGH
                            hostNetwork: true
                        

                        
spec:
  hostNetwork: true   # Node network access, IMDS reachable

                        
Fix: Remove hostNetwork, set IMDSv2 hop limit=1

                    


                    

                        

                            CRITICAL
                            hostPath: /
                        

                        
volumes:
  - hostPath:
      path: /          # Read/write entire host filesystem

                        
Fix: Use PersistentVolumeClaims, restrict via Pod Security Admission

                    


                    

                        

                            CRITICAL
                            SA cluster-admin
                        

                        
ClusterRoleBinding:
  roleRef: cluster-admin   # Full K8s API control

                        
Fix: Least-privilege RBAC, namespace-scoped Role/RoleBinding
