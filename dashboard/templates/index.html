K8s Container Escape Demo

                    
This demo illustrates a full attack chain exploiting Kubernetes misconfigurations, from initial RCE to cluster takeover, and shows how Cortex XDR detects and responds to each stage.

                


                

                    

                        

                            1
                            Infrastructure Setup
                        

                        
Deploy an EKS cluster on AWS with Terraform using admin credentials (one-time bootstrap). Terraform creates a dedicated dashboard IAM User with permanent Access Key credentials and a scoped dashboard operator IAM Role with least-privilege permissions (EKS, ECR, S3, Lambda, VPC, IAM). After deployment, use the dashboard user's permanent credentials — the dashboard automatically assumes the operator role via STS AssumeRole.

                        
Build a vulnerable Spring Boot application (CVE-2022-22965) as a Docker image and push it to ECR. Deploy it as a privileged pod with dangerous security settings:

                        
    
                            
  • privileged: true - full access to host devices
    • 
                            
  • hostPID: true - visibility into host processes
    • 
                            
  • hostPath: / - read/write access to host filesystem
    • 
                            
  • cluster-admin ServiceAccount - full K8s API access
    • 
                        

                    

                


                

                    

                        

                            2
                            Attack Chain
                        


                        

                            
Step 1: Spring4Shell RCE (CVE-2022-22965)

                            
Exploit the Spring4Shell vulnerability to manipulate Tomcat's AccessLogValve via the classLoader. This writes a JSP webshell to the application directory, providing remote command execution on the container.

                            
MITRE: T1190 (Exploit Public-Facing Application) / T1505.003 (Web Shell)

                        


                        

                            
Step 2: Container Escape

                            
Leverage the pod's privileged configuration to escape the container:

                            
    
                                
  • hostPID - access host process tree via /proc
    • 
                                
  • hostPath / - read host filesystem (kubelet creds, SSH keys, etc.)
    • 
                                
  • nsenter - enter host namespaces to execute commands as root on the node
    • 
                                
  • IMDS - access AWS Instance Metadata to steal IAM credentials
    • 
                            

                            
MITRE: T1611 (Escape to Host) / T1552 (Unsecured Credentials)

                        


                        

                            
Step 3: Cluster Takeover

                            
Use the pod's cluster-admin ServiceAccount token to gain full control of the Kubernetes cluster: list all namespaces, pods, secrets, nodes. Steal AWS credentials via IMDS for lateral movement to cloud services.

                            
MITRE: T1078.004 (Cloud Accounts) / T1552.007 (Container API)

                        

                    

                


                

                    

                        

                            3
                            Cortex XDR Detection & Response
                        

                        
Cortex XDR agent deployed on the cluster detects each stage of the attack:

                        
    
                            
  • Anti-Webshell - detects JSP webshell creation
    • 
                            
  • Container Escape Protection - detects nsenter and namespace breakout
    • 
                            
  • Behavioral Threat Protection - detects suspicious process activity
    • 
                            
  • Credential Gathering - detects access to sensitive files and IMDS
    • 
                        

                        
A prevention policy (rules + profiles) is imported on the Cortex tenant and assigned to the EKS endpoint group for K8s-specific protection.

                        
An automated containment playbook triggers on XDR issues. It uses a dedicated Cortex IAM User (permanent Access Key) to assume the lambda-invoker IAM Role via STS AssumeRole (SigV4-signed), then invokes the containment Lambda, which orchestrates: evidence collection, network isolation, RBAC revocation, scale down, node cordoning, and pod termination.
